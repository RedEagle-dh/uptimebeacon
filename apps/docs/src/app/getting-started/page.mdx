## üë§ User Management

### Creating Your Account

On first installation, you'll need to create an account:

1. Navigate to the login page
2. Click "Sign Up" or "Create Account"
3. Enter your email and password
4. Verify your email (if required)

**First user becomes admin automatically!**

### User Roles

UptimeBeacon has three user roles:

- **Admin**: Full access to all features and settings
- **User**: Can create monitors and view all data
- **Viewer**: Read-only access to monitors and status pages

## üìä Creating Your First Monitor

### HTTP/HTTPS Monitor

Monitor a website or API endpoint:

<Steps>
### Navigate to Monitors

Go to **Dashboard ‚Üí Monitors** and click "Create Monitor"

### Configure Basic Settings

```
Name: My Website
Type: HTTP/HTTPS
URL: https://example.com
Check Interval: 5 minutes
```

### Advanced Options (Optional)

- **Request Method**: GET, POST, HEAD, etc.
- **Request Headers**: Add custom headers
- **Expected Status Code**: 200, 201, etc.
- **Response Body Contains**: Check for specific text
- **Follow Redirects**: Enable/disable redirect following
- **SSL Certificate Verification**: Validate SSL certificates

### Save and Enable

Click "Create Monitor" - checks will start automatically!
</Steps>

### TCP Port Monitor

Monitor if a port is accessible:

```
Name: Database Server
Type: TCP
Host: db.example.com
Port: 5432
Check Interval: 2 minutes
```

### DNS Monitor

Verify DNS resolution:

```
Name: Domain DNS Check
Type: DNS
Hostname: example.com
Record Type: A
Expected Value: 93.184.216.34 (optional)
Check Interval: 15 minutes
```

### Ping Monitor

Test basic connectivity:

```
Name: Server Connectivity
Type: PING
Host: server.example.com
Packet Count: 4
Check Interval: 1 minute
```

## üîî Setting Up Notifications

Notifications alert you when monitors go down or recover.

### Email Notifications

<Steps>
### Create Notification Channel

Go to **Dashboard ‚Üí Notifications** and click "Add Channel"

### Configure Email

```
Channel Type: Email
Name: Team Alerts
Email Address: alerts@example.com
```

### Link to Monitors

Select which monitors should use this notification channel
</Steps>

### Slack Notifications

<Steps>
### Create Slack Webhook

1. Go to Slack ‚Üí Apps ‚Üí Incoming Webhooks
2. Create a new webhook
3. Copy the webhook URL

### Add to UptimeBeacon

```
Channel Type: Slack
Name: #engineering-alerts
Webhook URL: https://hooks.slack.com/services/YOUR/WEBHOOK/URL
```
</Steps>

### Discord Notifications

```
Channel Type: Discord
Name: Server Monitoring
Webhook URL: https://discord.com/api/webhooks/YOUR_WEBHOOK_ID/YOUR_TOKEN
```

### Telegram Notifications

<Steps>
### Create Bot with BotFather

1. Message @BotFather on Telegram
2. Use `/newbot` command
3. Get your bot token

### Get Chat ID

1. Add bot to your group
2. Send a message
3. Visit: `https://api.telegram.org/bot<YOUR_BOT_TOKEN>/getUpdates`
4. Find your chat ID

### Configure in UptimeBeacon

```
Channel Type: Telegram
Name: Operations Team
Bot Token: YOUR_BOT_TOKEN
Chat ID: YOUR_CHAT_ID
```
</Steps>

### Webhook Notifications

For custom integrations:

```
Channel Type: Webhook
Name: Custom Integration
Webhook URL: https://your-api.com/webhook
HTTP Method: POST
Headers: (optional custom headers)
```

Webhook payload format:
```json
{
  "monitor": {
    "id": "123",
    "name": "My Website",
    "url": "https://example.com"
  },
  "incident": {
    "status": "DOWN",
    "message": "Connection timeout",
    "timestamp": "2024-12-04T12:00:00Z"
  }
}
```

## ‚úÖ Best Practices

### Monitor Configuration

- **Check Intervals**:
  - Critical services: 1-5 minutes
  - Important services: 5-15 minutes
  - Less critical: 30 minutes - 1 hour

- **Timeout Settings**:
  - Fast endpoints: 5-10 seconds
  - Slower services: 15-30 seconds
  - External APIs: 30-60 seconds

### Notification Strategy

- **Critical Services**: Multiple channels (email + Slack/Discord)
- **Important Services**: Primary channel (email or chat)
- **Non-Critical**: Email only or batched notifications

### Status Page Setup

- Keep it simple and clear
- Group related services
- Show historical uptime data
- Enable incident subscriptions

## üí° Tips & Tricks

### Performance Optimization

- **Group similar monitors**: Use tags to organize
- **Stagger check intervals**: Avoid checking everything at once
- **Use appropriate timeouts**: Not too short, not too long

### Alert Fatigue Prevention

- **Set up retries**: Avoid false positives
- **Use escalation**: Start with warnings before critical alerts
- **Batch notifications**: Group multiple alerts when appropriate

### Team Collaboration

- **Define on-call schedule**: Who gets notified when
- **Document runbooks**: Link to resolution guides in monitors
- **Review incidents**: Learn from downtime events

## ‚ùì Common Questions

<details>
<summary>How often can I check my monitors?</summary>

Minimum interval is 30 seconds. For most use cases, 1-5 minutes is recommended to balance accuracy with resource usage.

</details>

<details>
<summary>Can I monitor services behind a firewall?</summary>

UptimeBeacon needs internet access to your services. For internal monitoring, you can:
- Expose a health check endpoint
- Use VPN or tunneling (Tailscale, Cloudflare Tunnel)
- Deploy UptimeBeacon inside your network

</details>

<details>
<summary>What happens when a monitor goes down?</summary>

1. UptimeBeacon performs retry checks (configurable)
2. If still down, creates an incident
3. Sends notifications to configured channels
4. Updates status page
5. Continues monitoring until service recovers
6. Closes incident and sends recovery notification

</details>

<details>
<summary>How do I monitor APIs with authentication?</summary>

Use custom headers in HTTP monitors:

```
Headers:
Authorization: Bearer YOUR_TOKEN
X-API-Key: YOUR_API_KEY
```

</details>

<details>
<summary>Can I get notified only during business hours?</summary>

Not currently supported, but on the roadmap! You can:
- Use different notification channels for different times
- Set up a webhook that filters by time
- Configure monitoring intervals differently

</details>
